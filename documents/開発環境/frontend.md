# Frontend

以下の表では **「言語」** として  
- JavaScript（ESNext）  
- TypeScript（型付きスーパーセット）  
- Elm / ReasonML / ReScript など関数型／型安全系言語  
- ClojureScript, Kotlin/JS, Dart, Rust, Python など、実際にブラウザ上で動作する他言語  

をそれぞれ列挙し、各言語が持つ代表的な **フレームワーク／ライブラリ** を網羅しています。  
※「メリット」「デメリット」は主にパフォーマンス・開発体験・エコシステムの観点から抜粋しました。  

| 言語 | フレームワーク / ライブラリ | 主な特徴・用途 | メリット (主) | デメリット (主) |
|------|---------------------------|---------------|--------------|-----------------|
| **JavaScript** | React | コンポーネント指向、JSX、豊富なエコシステム（Redux, Next.js 等） | ・大規模コミュニティ<br>・SSR/SSG が可能 (Next.js)<br>・再利用性が高い | ・ビルドサイズが大きくなることも<br>・学習曲線（Hooks, Context など） |
| **JavaScript** | Vue 3 / Nuxt.js | Composition API、テンプレート構文、SSR/SSG が標準 (Nuxt) | ・直感的なテンプレート構文<br>・型安全化が容易 (Vue + TypeScript)<br>・サーバー側レンダリングが簡単 | ・公式エコシステムが小さい（React ほど）<br>・大型プロジェクトでのパフォーマンスチューニングは手間 |
| **JavaScript** | Angular | 完全フレームワーク、DI、モジュール化、TypeScript 標準 | ・包括的な機能セット<br>・大規模開発に適した構造<br>・公式サポートが充実 | ・学習曲線が急で重い<br>・ビルドサイズと起動時間が長め |
| **JavaScript** | Svelte / SvelteKit | コンパイラ型フレームワーク、ランタイムオーバーヘッドほぼゼロ | ・高速実行（最適化済）<br>・シンプルな構文<br>・ビルドサイズが小さくなる | ・エコシステムがまだ拡大途上<br>・既存ライブラリの統合に手間 |
| **JavaScript** | SolidJS | 高速リアクティブ、JSX ベース | ・最適化された再描画<br>・React 互換性あり（コード移行が容易）<br>・軽量 | ・コミュニティ規模が小さくドキュメント不足 |
| **JavaScript** | Preact | React API 互換、ファイルサイズを 3〜5 kB に抑制 | ・React コードベースで軽量化<br>・SSR/SSG が可能（Preact CLI, Vite 等） | ・一部 React 専用プラグインは動かない |
| **JavaScript** | Alpine.js | jQuery 風構文、軽量なリアクティブフレームワーク | ・既存プロジェクトへの導入が容易<br>・ビルドサイズが非常に小さい | ・大規模 SPA 用には不向き |
| **JavaScript** | Lit / LitElement (Web Components) | Web Components をベースにした軽量フレームワーク | ・ブラウザ標準を利用、再利用性高い<br>・複数フレームワークと混用可能 | ・学習コスト（テンプレートタグ）<br>・エコシステムが限定的 |
| **JavaScript** | Stencil | Web Components + TypeScript の生成ツール | ・小さなビルドサイズ<br>・他フレームワークと共存しやすい | ・学習曲線がある（構成ファイル） |
| **JavaScript** | Qwik | 「遅延ロード」型、再描画を最小化 | ・高速な初期表示<br>・大規模 SPA でのパフォーマンス向上 | ・新しい概念に慣れる必要がある |
| **JavaScript** | Astro | コンポーネントベース、SSR/SSG 重視 | ・サーバーだけで HTML を生成し、クライアントは最小化<br>・多フレームワーク共存（React, Vue 等） | ・SPA では機能が限定的 |
| **JavaScript** | Remix / Next.js / Nuxt.js (SSR/SSG 専用) | フルスタック SSR／SvelteKit など | ・SEO 対策が容易<br>・データフェッチングの統合が便利 | ・ビルド時間やランタイムサイズが大きい |
| **TypeScript** | Angular | TypeScript 標準で書かれたフレームワーク | ・型安全性が高く、開発時に多くのエラーを検出<br>・包括的な機能セット | ・学習曲線が急で重い |
| **TypeScript** | React (TSX) | JSX + 型定義を併用 | ・大規模プロジェクトでも型チェックで安全性向上<br>・既存 JavaScript コードとの混在が容易 | ・型定義ファイルの管理が必要 |
| **TypeScript** | Vue 3 (Composition API) | TypeScript 対応が公式に組み込まれたバージョン | ・テンプレートと TS が自然に統合<br>・型安全性でバグ削減 | ・Vuex の型定義はやや複雑 |
| **TypeScript** | SvelteKit (TS) | コンパイラ時に TypeScript をチェック | ・Svelte の高速性 + TS の安全性を両立 | ・一部ライブラリが TS 対応でないことも |
| **TypeScript** | SolidJS (TS) | TypeScript 完全対応 | ・型推論が優秀<br>・コンパイル時にエラーを検出 | ・公式ドキュメントはまだ発展途上 |
| **TypeScript** | Preact TS | 型付きバージョンで軽量化 | ・React 互換性＋TS の安全性 | ・型定義が完全ではないケースも |
| **Elm** | Elm | 型安全な関数型言語、コンパイル時エラー防止 | ・バグが極めて少ない<br>・コード保守性高い | ・学習曲線が急で既存ライブラリとの統合が難しい |
| **ClojureScript** | Reagent / Rum | React ラッパー、関数型構造 | ・不変データで安全性高い<br>・コンパクトなコード | ・エコシステムは小さくドキュメント不足 |
| **ReasonML / ReScript** | ReScript (React, ReasonReact) | 型安全な JS superset、JSX 互換 | ・高速なビルドと実行<br>・型チェックが強力 | ・コミュニティ規模が限定的 |
| **Kotlin/JS** | Kotlin Multiplatform / Kotlin React | JVM の堅牢さをフロントに持ち込む | ・統一されたコードベースでバックエンドと共有可能<br>・型安全性高い | ・ビルドツールがやや複雑 |
| **Dart** | Flutter Web / AngularDart | クロスプラットフォーム UI（Flutter）やフロント専用 AngularDart | ・ネイティブに近いパフォーマンス<br>・デザインの一貫性 | ・Web 専用機能が限定的、ビルドサイズが大きい |
| **Rust** | Yew / Seed | WebAssembly を利用した高速フロント | ・安全な言語でメモリ管理不要<br>・非常に高性能 | ・WASM のロード時間が課題<br>・学習曲線が急 |
| **Python** | Transcrypt / Brython | Python をブラウザ側で実行 | ・既存 Python コードを再利用可能 | ・パフォーマンスやエコシステムは限定的 |
| **Go** | GopherJS / Vecty | Go でフロント開発 | ・型安全、ツールチェーンが充実 | ・ビルドサイズが大きく、コミュニティは小さい |

## 使い分けのポイント

1. **大規模な業務アプリ**  
   - Angular (TypeScript 標準) / React + TSX（型安全で既存 JS と併用しやすい）  

2. **高速開発・小〜中規模プロジェクト**  
   - Vue 3（TS 推奨）、SvelteKit、SolidJS  

3. **軽量化が最重要**  
   - Preact, Alpine.js, Lit / Stencil  

4. **パフォーマンス重視の SPA**  
   - Qwik, SvelteKit, SolidJS  

5. **静的サイトやマルチフレームワーク混在**  
   - Astro、Next.js（SSR/SSG）  

6. **型安全・関数型志向**  
   - Elm、ReScript、ClojureScript、Rust (Yew)  

7. **クロスプラットフォーム UI が必要**  
   - Flutter Web（Dart）  

8. **既存バックエンドとコードを共有したい**  
   - Kotlin Multiplatform, ReasonML / ReScript

# ビルド／開発サーバー（バンドラ）を網羅した表

| ツール | 主な特徴・用途 | メリット (主) | デメリット (主) |
|--------|----------------|--------------|-----------------|
| **Vite** | ES‑M モジュールをそのままサーバーに流し、`esbuild` で高速ビルド。React / Vue / Svelte 等フレームワークとネイティブに統合。 | ・開発時の起動・ホットリロードが数ミリ秒<br>・最適化済みビルドは Rollup を利用、プラグインエコシステム豊富 | ・古いブラウザでのサポートにはトランスパイルが必要<br>・大型プロジェクトでのプラグイン管理がやや煩雑 |
| **Webpack** | 最も広く使われるモジュールバンドラー。ローダーとプラグインでほぼ無限に拡張可能。 | ・成熟したエコシステム<br>・大規模なカスタマイズが可能（環境変数、コード分割、ホットリロードなど） | ・設定が複雑になりやすい<br>・ビルド時間が長くなることも |
| **Rollup** | ES モジュールを前提にしたバンドラー。シングルファイルライブラリ向けに最適化されている。 | ・出力が小さく、ツリーシェイキングが優秀<br>・ビルド速度が速い | ・SPA の開発にはプラグインや設定が必要で、React/Vue などのサポートは Webpack より限定的 |
| **Snowpack** | Vite 前に登場した「モジュールをそのままブラウザへ」型のビルドツール。ESM を直接利用し、`npm install` のみで即座に起動。 | ・非常に高速な開発サーバー<br>・小規模〜中規模プロジェクト向き | ・現在は Vite に取って代わられているためエコシステムが縮退 |
| **Parcel** | 「設定不要」で即座にバンドル開始できるフレームワークレスツール。自動コード分割、トランスパイルを内蔵。 | ・ゼロコンフィグで始められる<br>・プラグインは比較的簡単に追加可能 | ・ビルドサイズがやや大きくなることがある<br>・大規模アプリでは設定の柔軟性が不足 |
| **esbuild** | Go で書かれた高速な JavaScript / TypeScript バンドラー & トランスパイラ。Vite の開発サーバーはこのエンジンを利用している。 | ・ビルド時間が極めて短い（数ミリ秒）<br>・CLI だけで軽量に動作 | ・機能はまだ発展途上（プラグインエコシステムが小さい） |
| **Bun** | JavaScript 実行環境 + パッケージマネージャ + バンドラー。Vite よりもさらに高速で、CLI が統合されている。 | ・npm 互換の `bun install` が速い<br>・開発サーバーとバンドルが同一ツールで完結 | ・まだ成熟段階にあり、エコシステムは限定的 |

### 選び方のポイント
| シナリオ | 推奨ツール |
|----------|------------|
| **高速な開発体験**（SPA 開発） | Vite / esbuild を利用したプロジェクト構築 |
| **既存 Webpack 設定を再利用** | そのまま Webpack を継続、または Vite のプラグインで移行 |
| **小規模・ゼロ設定開発** | Parcel / Snowpack（ただし今は Vite が主流） |
| **ライブラリ作成・Tree‑Shaking 重視** | Rollup |
| **統合された実行環境と極めて高速ビルド** | Bun |

これらを参考に、プロジェクトの規模・要件・チームスキルに合わせて最適なツールを選択してください。
